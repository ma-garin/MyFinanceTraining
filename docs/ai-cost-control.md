# AIコスト管理

## 方針

AIは常時利用しない。通常処理はルールベースで行い、必要な箇所だけAIを使う。

```text
通常走行：ルールベース
難所だけ：AI
```

## AIを使う場面

| 場面 | 方針 |
|---|---|
| ニュース分類 | 原則ルールベース |
| 既存テンプレート照合 | ルールベース |
| 連想ツリーの初期候補生成 | 必要時のみAI |
| 3〜5段階目の深掘り | AI利用候補 |
| 失敗条件の洗い出し | AI利用候補 |
| 反対シナリオ生成 | AI利用候補 |
| 銘柄候補抽出 | 原則ルールベース |
| バックテスト | コードで実行 |

## AI呼び出し条件

AIは以下の場合のみ呼ぶ。

- 未登録テーマを扱う
- 既存テンプレートで候補が少ない
- 3段階目以降の連想が弱い
- 失敗条件が不足している
- ユーザーが明示的に深掘りを実行した

## コスト制御

必須機能：

- 1回あたりの最大出力量を制限する
- 1日あたりのAI実行回数を制限する
- 月あたりのAI実行回数を制限する
- 同一入力の結果を再利用する
- AI実行履歴を保存する
- AI連携なしでも基本機能を使える

## 推奨API運用

初期版では、事前購入または上限管理しやすいAPIを優先する。

| 候補 | 用途 |
|---|---|
| Gemini API | 連想ツリーの大量生成候補 |
| OpenAI API | 品質比較、深い仮説レビュー候補 |
| Claude API | 初期は手動壁打ち、API実装は後回し |

## 実行単位

AIリクエストは小さく分割する。

悪い例：

```text
このニュースから投資先を全部考えて
```

良い例：

```text
このイベントから3〜5段階目の生活者行動変化だけを10個出す
```

## キャッシュ方針

同じニュースや同じイベントでは再生成せず、過去の結果を再利用する。

```text
model + promptType + normalizedInput + templateVersion
```

## AIプロンプト種別

| 種別 | 目的 |
|---|---|
| association_expand | 連想ツリーの分岐生成 |
| counter_scenario | 失敗条件/反対シナリオ生成 |
| hypothesis_naming | 仮説ID命名 |
| missing_angle | 見落とし観点の提示 |
| theme_mapping | 日本株テーマへの変換 |

## 禁止事項

- AIに売買判断を任せる
- AI出力をそのまま推奨銘柄扱いする
- すべてのニュースで自動実行する
- バックテストをAIの文章推測だけで代替する
- 機密情報をリポジトリに保存する

## 目標

最初の目標は、月数百円から千円以下で運用できる設計にすること。
